01 Overview

A tool used every day that made people's jobs harder.

During my internship at LuminateOne, I was given the Automote redesign as my core project. Automote is a stock and inventory management platform used daily by car dealership staff to track orders, manage vehicle builds, and monitor inventory.

The existing interface had accumulated years of fixes on top of fixes. My job was to redesign the core workflows from scratch in Figma, producing a fully interactive prototype the development team could use to plan implementation.

B2B design is unforgiving. Dealership staff aren't browsing — they're working under time pressure, handling real financial transactions. The cost of confusion isn't frustration, it's mistakes with money and customer relationships.

02 Brief & User Analysis

Understanding who I was designing for.

The typical user: An admin who is knowledgeable and uses computers daily, but wouldn't understand developer environments. This creates problems when information is laid out in a way that isn't easily understandable.

The end goal was to have a system that the user can look at and instantly understand where they need to navigate — without clicking through several screens to figure out what headings mean. It also needed to be clearer what was being displayed to end users/customers, as there was potential to add things without realising if they were published or not.

The Brief

Re-design the inventory manager for Hyundai's web applications. I was given free rein in terms of direction and design process — a level of trust that meant I had to be rigorous in justifying every decision.

Core Requirements

  • Add a landing page with clear navigation
  • Add content filters on every page with sort functionality
  • Organise content in logical, labelled groups
  • Enable bulk selection and editing
  • Clarify what information is customer-facing vs internal
03 UI Audit

Annotating what was broken.

Before designing anything, I annotated the existing interface to identify specific issues. This wasn't about aesthetics — it was about understanding where the interface failed the user.

Issue identification: navigation complexity, unclear labelling, and missing hierarchy.

Further issues: content organisation, visibility settings, and bulk action limitations.

Key findings from the audit: Labels didn't need to all be displayed at once. Trim level labels didn't need to include the full model name. Photos should sit left-aligned for better visual flow. Some information could be hidden under accordions closed by default. Clear labelling was needed to distinguish customer-facing content from internal notes.

04 Wireframing

Sketching the structure before the pixels.

Based on the audit, I created wireframe sketches for nearly every element in the interface — selecting, editing, filtering, and organising content.

Initial filter concept — a filter button approach.

Refined filter — persistent panel rather than modal, based on mentor feedback.

Mentor feedback that changed my approach: My initial sketches included a filter button, but my mentor pushed me to think more deeply about how the user would actually interact with it. What type of filter made sense? Should it be a dropdown, a sidebar, or something else? This forced me to break each component down — search, filter, tags — and consider interaction before combining them into full pages.

08 Final Delivered Design

The redesigned platform.

After iterating through drafts and incorporating mentor feedback, the final version delivered on all core requirements.

Orders management — final. Filterable by ID, customer, model, date, and production status. Clear hierarchy, consistent row actions, scannable status indicators.

Order detail and editing — final. Inline status updates with confirmation states. No more navigating away to make a simple change.

Inventory dashboard — final. High-level overview of stock, incoming orders, and vehicles in production.

09 The Problems Solved

Six problems that made a hard job harder.

Dealership staff use Automote for hours every day. Every friction point compounds across hundreds of interactions.

  • 01
    Text and buttons cut off

    UI elements truncated at screen edges. Labels and data values disappeared mid-character — staff couldn't read critical information without resizing windows.

  • 02
    Navigation required too many steps

    Too many clicks to reach common actions. Users had to memorise locations rather than discover them.

  • 03
    No visual CTA hierarchy

    Primary actions weren't distinguishable from secondary ones. Users hesitated, clicked wrong buttons, and had to undo actions.

  • 04
    Actions completed silently

    No confirmation states, no error messages, no loading indicators. Users couldn't tell whether something had worked or failed.

  • 05
    Layout broke on different screens

    Built for one fixed resolution. On any other display, components collapsed or overlapped unpredictably.

  • 06
    No consistent design language

    Spacing, type sizes, button styles, and colours varied across screens — making the platform feel unfinished and untrustworthy.

10 Key Design Decisions

The calls I made and why.

🗂

Two-panel configuration

Details and configuration steps in a persistent split view. Dealers never lose context.

🔍

Collapsible side filter

Visible when needed, out of the way when not. Filters by ID, customer, model, date, and status.

🚦

Status as clear text

Production status displayed as readable text labels, not icons or colour alone. Self-explanatory.

⚓

Structured sidebar nav

Tools grouped into logical categories. Dealers always know where they are.

🎯

Primary / secondary hierarchy

Primary actions use filled buttons. Destructive actions are subdued — reducing errors.

📐

Unified component system

Spacing scale, type hierarchy, button variants — applied consistently. Consistency is a feature.

11 Testing + Iteration

What the review sessions revealed.

Stakeholder review

Status column wasn't scannable

Dealers scanning a long orders list couldn't quickly identify which orders needed action.

What I changed

Added visual weight to action-required statuses

Introduced subtle colour coding. Text labels retained — colour was additive, not the sole indicator.

Stakeholder review

Bulk actions were missing

Sales managers needed to update multiple orders at once. v1 required individual updates.

What I changed

Designed a bulk selection and action system

Added row-level checkboxes and a contextual bulk action bar that appeared when multiple items were selected.

Constraints

  • Internship timeline — six months. Focused on highest-frequency workflows
  • No direct access to end users — feedback came through LuminateOne stakeholders
  • Prototype for planning purposes only — edge cases needed to be thorough
  • Existing branding constraints — design had to work within Automote's established palette

13 Midway Reflection

What the internship taught me about real-world UX.

From the process of getting from brief to completed Figma prototype, I learned a lot — particularly about refining my UX skills in a real commercial environment.

My first drafts focused too much on making the interface look good without enough consideration for how the user would actually interact with each element. Using the feedback I got from my mentor, I broke each component down — search, filter, tags — and thought about how the user would interact with these elements before combining them to create the actual pages.

It was interesting to see how the theory work from all the classes I had taken — creating personas, wireframing, interaction design — actually applied in a real-world setting. The gap between academic exercises and commercial work is real, but the foundation matters. The process works if you actually use it.

The biggest shift was learning to prioritise interaction over aesthetics. A beautiful interface that's hard to use is still a failure. A functional interface that's clear and efficient — that's what users actually need.
14 Final Reflection

Designing for people at work.

B2B design is unforgiving. Consumer apps get forgiven for being confusing — people close them and move on. A tool you use for eight hours a day is different.

This project taught me that context of use changes everything. Dealership staff aren't browsing — they're working under time pressure, handling real financial transactions. The cost of confusion isn't frustration, it's mistakes with money and customer relationships.

The most valuable lesson was that consistency is a feature. In a complex tool used daily, predictability reduces cognitive load more than any clever interaction. Users should be thinking about their work, not the interface.

Not having direct access to end users was a real constraint — something I'd push back on in future. Even one session watching a dealer use the existing system would have been worth more than hours of assumption.
