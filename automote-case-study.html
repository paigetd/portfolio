01 — Overview

A tool used every day by car dealers that made their jobs harder.

During my internship at LuminateOne, I was given the Automote redesign as my core project — a stock and inventory management platform used daily by car dealership staff to track orders, manage vehicle builds, and monitor inventory.

The existing interface had accumulated years of fixes on top of fixes. The result was a tool that was genuinely painful to use. My job was to redesign the core workflows from scratch in Figma, producing a fully interactive prototype that the development team could use to plan implementation.

02 — The Problem

Six problems that made a hard job harder.

Dealership staff use Automote for hours every day. Managing dozens of orders, tracking vehicles through production pipelines, configuring complex builds for customers. Every friction point compounds across hundreds of interactions.

Before designing anything, I spent time understanding the actual workflows — what tasks were done most frequently, where users were getting stuck, and what the existing UI was forcing them to work around rather than with.

  • 01
    Text and buttons cut off

    UI elements were truncated at screen edges with no overflow handling. Labels, CTAs, and data values disappeared mid-character.

  • 02
    Navigation required too many steps

    Too many clicks to reach common actions. The hierarchy was flat and unlabelled — users had to memorise locations rather than discover them.

  • 03
    No visual CTA hierarchy

    Primary actions weren't distinguishable from secondary ones. Everything looked the same weight, which made it unclear what to do next.

  • 04
    Actions completed silently

    No confirmation states, no error messages, no loading indicators. Users couldn't tell whether something had worked or failed.

  • 05
    Layout broke on different screens

    Built for one fixed resolution. On any other display, components collapsed or overlapped unpredictably.

  • 06
    No consistent design language

    Spacing, type sizes, button styles, and colours varied across screens — making the platform feel unfinished and untrustworthy.

03 — Core User Flow

Mapping the most critical workflow.

Order management was the highest-frequency task. I mapped it first because fixing this flow would have the biggest daily impact for users.

View orders list
Filter by status / model
Select order
View / edit details
Update production status
Confirmation state

Previously this flow had no filter system, no inline editing, and no confirmation. Every status update required navigating away and back.

05 — Final Design

The redesigned platform.

Order Management

Filterable by ID, customer, model, date, and production status. Clear hierarchy, consistent row actions, and a status system that communicates where each order actually is.

Vehicle Configuration

Two-panel layout keeps build details and configuration steps in view at the same time — removing the need to navigate back and forth mid-task.

06 — Key Design Decisions

The calls I made and why.

🗂

Two-panel layout for configuration

The configure builds screen splits details and configuration steps into a persistent two-panel view. Dealers no longer lose context when adding or changing build options.

🔍

Collapsible side filter

Filter panel is visible when needed and out of the way when not. Filters by ID, customer, model, date range, and production status — all the dimensions dealers actually work with.

🚦

Status as clear text, not icons

Production status is displayed as readable text labels, not icons or colour alone. No legend required — the system is self-explanatory at a glance.

⚓

Structured sidebar navigation

Tools grouped into logical categories (Data Manager, eCommerce, Vehicles) with persistent context. Dealers always know where they are and how to get somewhere else.

🎯

Clear primary / secondary action hierarchy

Primary actions use filled buttons with strong colour. Secondary and destructive actions (edit, delete) are iconographic and subdued — reducing accidental actions in high-pressure workflows.

📐

Unified component system in Figma

Built a consistent component library — spacing scale, type hierarchy, button variants, form elements — applied across every screen. Consistency is a feature in a complex daily-use tool.

07 — Testing & Iteration

What the review sessions revealed.

Stakeholder review feedback

The status column wasn't immediately scannable

During a review session with the LuminateOne team, it became clear that dealers scanning a long orders list couldn't quickly identify which orders needed action. All statuses looked visually equivalent.

What I changed

Added visual weight to action-required statuses

Introduced subtle colour coding for statuses requiring dealer action (e.g. "Awaiting Confirmation" in amber, "Ready for Pickup" in green). Text labels were retained — colour was additive, not the sole indicator.

Stakeholder review feedback

Bulk actions were missing from the orders view

Sales managers needed to update multiple orders at once — for example, marking a batch as "In Transit" after a delivery run. The v1 design required updating each order individually.

What I changed

Designed a bulk selection and action system

Added row-level checkboxes and a contextual bulk action bar that appeared when multiple items were selected. Prototyped the full interaction in Figma for the dev handoff.

Constraints I worked within

  • Internship timeline — six months meant scoping realistically. I focused on the highest-frequency workflows and clearly marked lower-priority screens as out of scope for the prototype
  • No direct access to end users (dealership staff) — all feedback came through LuminateOne stakeholders, which introduced a layer of interpretation
  • The prototype was for planning purposes only — I wasn't involved in implementation, which meant edge cases and error states needed to be thorough in the handoff documentation
  • Existing branding constraints meant the visual design had to work within Automote's established colour palette
08 — Outcomes

How the work was received.

Positive stakeholder reception at review

The LuminateOne team confirmed the redesign addressed the core navigation and clarity issues they'd been hearing about from dealership staff. The prototype was accepted as the basis for development planning.

Reduced cognitive load in high-frequency workflows

The two-panel configure screen and the collapsible filter system were specifically called out as solving problems dealers had raised. Status legibility was noted as a clear improvement over the previous version.

Comprehensive handoff for development

The fully interactive Figma prototype covered all core workflows with annotated component specs, interaction states, and documented design decisions — reducing ambiguity for the development team.

09 — Reflection

Designing for people at work.

"B2B design is unforgiving. Consumer apps get forgiven for being confusing — people close them and move on. A tool you use for eight hours a day is different. Every rough edge costs you something."

This project taught me that context of use changes everything. Dealership staff aren't browsing — they're working under time pressure, handling real financial transactions, often across multiple screens. The cost of confusion isn't frustration, it's mistakes with money and customer relationships on the line.

The most valuable lesson was that consistency is a feature. In a complex tool used daily, predictability reduces cognitive load more than any clever interaction. Users should be thinking about their work, not about the interface.

Not having direct access to end users was a real constraint — something I'd push back on in a future project. Stakeholder feedback is useful, but it's a layer removed from the people actually doing the work. Even one session watching a dealer use the existing system would have been worth more than hours of assumption.
