01 Overview

A tool used every day that made people's jobs harder.

During my internship at LuminateOne, I was given the Automote redesign as my core project. Automote is a stock and inventory management platform used daily by car dealership staff to track orders, manage vehicle builds, and monitor inventory.

The existing interface had accumulated years of fixes on top of fixes. My job was to redesign the core workflows from scratch in Figma, producing a fully interactive prototype the development team could use to plan implementation.

B2B design is unforgiving. Dealership staff work under time pressure with real financial transactions. The cost of confusion isn't frustration — it's mistakes with money and customer relationships.

02 The Problem

Six problems that made a hard job harder.

Every friction point in Automote compounded across hundreds of daily interactions. Here's what the audit surfaced.

Before — original interface

The original Automote interface. Text truncated at edges, no visual hierarchy, actions requiring multiple navigations away and back.

  • 01
    Text and buttons cut off

    UI elements truncated at screen edges. Labels, CTAs, and data values disappeared mid-character.

  • 02
    Navigation required too many steps

    Too many clicks to reach common actions. Hierarchy was flat — users had to memorise locations rather than discover them.

  • 03
    No visual CTA hierarchy

    Primary actions weren't distinguishable from secondary ones. Everything looked the same weight.

  • 04
    Actions completed silently

    No confirmation states, no error messages, no loading indicators. Users couldn't tell if something had worked or failed.

  • 05
    Layout broke on different screens

    Built for one fixed resolution. On any other display, components collapsed or overlapped unpredictably.

  • 06
    No consistent design language

    Spacing, type sizes, button styles varied across screens — making the platform feel unfinished and untrustworthy.

03 Core User Flow

Updating order status — from 5 steps to 2.

Order status updates were the most frequent dealer action. Previously required navigating away, making the change, then returning. Redesigned to inline editing with immediate confirmation.

View orders list
Filter by status / model
Select order
View / edit inline
Update production status
Confirmation state

Previously: no filter, no inline editing, no confirmation. Every update required navigating away and back.

05 Final Design — Orders

Order management, rebuilt.

The orders screen is where dealers spend most of their time. It needed to be filterable, scannable, and clear about which orders needed action — without requiring any memorisation.

Order management — filterable by ID, customer, model, date, and production status. Visual weight distinguishes action-required statuses from passive ones.

Orders list with filter panel open — collapsible sidebar keeps the primary content visible while filtering is active

Filter panel expanded — grouped by category, with clear visual hierarchy between filter types

Inline order editing — status can be updated without navigating away. Confirmation state appears immediately on save.

Inventory overview — consistent row structure, status indicators, and quick-action controls across all list views

06 Final Design — Vehicle Configuration

Configure builds without losing context.

The two-panel configuration screen was the key design decision. Dealers need to see the order summary and the configuration steps simultaneously — previously they had to navigate between them.

Two-panel vehicle configuration — order details persistent on the left, configuration steps on the right. No navigation required mid-task.

Trim level selection with visual preview

Adding configuration options — clear step progress indicator

Editing existing config — inline editing with clear save/cancel hierarchy

Configuration step flow — numbered progress keeps dealers oriented through complex multi-step builds

Initial high-fidelity exploration — testing the split-panel layout before committing to the final direction

07 Key Design Decisions

The calls I made and why.

🗂

Two-panel configuration

Details and configuration steps in a persistent split view. Dealers never lose context when building out a vehicle order.

🔍

Collapsible side filter

Visible when needed, out of the way when not. Filters by ID, customer, model, date range, and production status.

🚦

Status as clear text

Production status as readable text labels, not icons alone. No legend required — self-explanatory at a glance.

⚓

Structured sidebar nav

Tools grouped into logical categories with persistent context. Dealers always know where they are.

🎯

Primary / secondary hierarchy

Primary actions use filled buttons with strong colour. Destructive actions subdued — reducing accidental irreversible actions.

📐

Unified component system

Spacing scale, type hierarchy, button variants, form elements — applied consistently. Consistency is a feature.

08 Testing + Iteration

What the review sessions revealed.

Stakeholder review

Status column wasn't scannable

Dealers scanning a long orders list couldn't quickly identify which orders needed action. All statuses looked visually equivalent.

What I changed

Added visual weight to action-required statuses

Introduced subtle colour coding for statuses requiring dealer action. Text labels retained — colour additive, not the sole indicator.

Stakeholder review

Bulk actions were missing

Sales managers needed to update multiple orders at once. v1 required updating each order individually.

What I changed

Designed a bulk selection and action system

Row-level checkboxes and a contextual bulk action bar appearing when multiple items are selected. Full interaction prototyped in Figma.

Constraints

  • Internship timeline — six months, so focused on highest-frequency workflows
  • No direct access to end users — all feedback came through LuminateOne stakeholders
  • Prototype for planning only — not involved in implementation, so edge cases needed to be thorough
  • Existing branding constraints meant working within Automote's established colour palette
09 Outcomes

How the work was received.

Pain points confirmed solved

LuminateOne confirmed the redesign addressed navigation and clarity issues previously raised by dealership staff. Approved as the basis for development without further revision.

Reduced cognitive load specifically called out

The two-panel configure screen and collapsible filter system were specifically noted as solving problems dealers had raised. Status legibility noted as a clear improvement.

Prototype reduced dev ambiguity

Fully interactive prototype with annotated specs, interaction states, and documented decisions — enabling clearer implementation planning and reduced design-dev back-and-forth.

10 Reflection

Designing for people at work.

B2B design is unforgiving. Consumer apps get forgiven for being confusing. A tool you use for eight hours a day is different.

This project taught me that context of use changes everything. Dealership staff work under time pressure handling real financial transactions. The cost of confusion isn't frustration — it's mistakes with money and customer relationships.

The most valuable lesson was that consistency is a feature. In a complex tool used daily, predictability reduces cognitive load more than any clever interaction. Users should be thinking about their work, not the interface.

Not having direct access to end users was a real constraint — something I'd push back on in future. Even one session watching a dealer use the existing system would have been worth more than hours of assumption.
