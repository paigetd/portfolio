01 Overview

A tool used every day that made people's jobs harder.

During my internship at LuminateOne, I was given the Automote redesign as my core project. Automote is a stock and inventory management platform used daily by car dealership staff to track orders, manage vehicle builds, and monitor inventory.

The existing interface had accumulated years of fixes on top of fixes. My job was to redesign the core workflows from scratch in Figma, producing a fully interactive prototype the development team could use to plan implementation.

B2B design is unforgiving. Dealership staff aren't browsing — they're working under time pressure, handling real financial transactions. The cost of confusion isn't frustration, it's mistakes with money and customer relationships.

02 The Problem

Six problems that made a hard job harder.

Dealership staff use Automote for hours every day. Managing dozens of orders, tracking vehicles through production pipelines, configuring complex builds. Every friction point compounds across hundreds of interactions.

  • 01
    Text and buttons cut off

    UI elements were truncated at screen edges with no overflow handling. Labels, CTAs, and data values disappeared mid-character.

  • 02
    Navigation required too many steps

    Too many clicks to reach common actions. Hierarchy was flat and unlabelled — users had to memorise locations rather than discover them.

  • 03
    No visual CTA hierarchy

    Primary actions weren't distinguishable from secondary ones. Everything looked the same weight — unclear what to do next.

  • 04
    Actions completed silently

    No confirmation states, no error messages, no loading indicators. Users couldn't tell whether something had worked or failed.

  • 05
    Layout broke on different screens

    Built for one fixed resolution. On any other display, components collapsed or overlapped unpredictably.

  • 06
    No consistent design language

    Spacing, type sizes, button styles, and colours varied across screens — making the platform feel unfinished and untrustworthy.

03 Core User Flow

Updating order status — from 5 steps to 2.

Order status updates were the most frequent dealer action. Previously required navigating away, making the change, then returning. Redesigned to inline editing with immediate confirmation.

View orders list
Filter by status / model
Select order
View / edit details
Update production status
Confirmation state

Previously: no filter system, no inline editing, no confirmation. Every status update required navigating away and back.

05 Final Design

The redesigned platform.

Two screens solved the most frequent dealer workflows. The orders list with its filter panel, and the two-panel vehicle configuration screen that keeps context always visible.

Order management — filterable by ID, customer, model, date, and production status. Clear hierarchy, consistent row actions, a status system that communicates where each order actually is.

Vehicle configuration — two-panel layout keeps build details and configuration steps in view simultaneously. No more navigating away mid-task.

06 Key Design Decisions

The calls I made and why.

🗂

Two-panel configuration

Details and configuration steps in a persistent split view. Dealers never lose context when adding or changing build options.

🔍

Collapsible side filter

Visible when needed, out of the way when not. Filters by ID, customer, model, date range, and production status.

🚦

Status as clear text

Production status displayed as readable text labels, not icons or colour alone. No legend required — self-explanatory at a glance.

⚓

Structured sidebar nav

Tools grouped into logical categories with persistent context. Dealers always know where they are and how to get somewhere else.

🎯

Primary / secondary hierarchy

Primary actions use filled buttons with strong colour. Destructive actions are subdued — reducing accidental irreversible actions.

📐

Unified component system

Spacing scale, type hierarchy, button variants, form elements — applied consistently across every screen. Consistency is a feature.

07 Testing + Iteration

What the review sessions revealed.

Stakeholder review

Status column wasn't scannable

Dealers scanning a long orders list couldn't quickly identify which orders needed action. All statuses looked visually equivalent.

What I changed

Added visual weight to action-required statuses

Introduced subtle colour coding for statuses requiring dealer action. Text labels retained — colour was additive, not the sole indicator.

Stakeholder review

Bulk actions were missing

Sales managers needed to update multiple orders at once. The v1 design required updating each order individually.

What I changed

Designed a bulk selection and action system

Added row-level checkboxes and a contextual bulk action bar that appeared when multiple items were selected. Full interaction prototyped in Figma.

Constraints

  • Internship timeline — six months meant scoping realistically. Focused on highest-frequency workflows
  • No direct access to end users — all feedback came through LuminateOne stakeholders
  • Prototype for planning purposes only — not involved in implementation, so edge cases needed to be thorough
  • Existing branding constraints meant the visual design had to work within Automote's established colour palette
08 Outcomes

How the work was received.

Key pain points confirmed solved

LuminateOne confirmed the redesign addressed navigation and clarity issues previously raised by dealership staff. Approved as the basis for development without further revision rounds.

Reduced cognitive load specifically called out

The two-panel configure screen and collapsible filter system were specifically noted as solving problems dealers had raised. Status legibility noted as a clear improvement.

Prototype reduced dev ambiguity

Fully interactive prototype with annotated specs, interaction states, and documented decisions. Enabled clearer implementation planning and reduced design-dev back-and-forth.

09 Reflection

Designing for people at work.

B2B design is unforgiving. Consumer apps get forgiven for being confusing — people close them and move on. A tool you use for eight hours a day is different.

This project taught me that context of use changes everything. Dealership staff aren't browsing — they're working under time pressure, handling real financial transactions. The cost of confusion isn't frustration, it's mistakes with money and customer relationships.

The most valuable lesson was that consistency is a feature. In a complex tool used daily, predictability reduces cognitive load more than any clever interaction. Users should be thinking about their work, not the interface.

Not having direct access to end users was a real constraint — something I'd push back on in future. Even one session watching a dealer use the existing system would have been worth more than hours of assumption.
